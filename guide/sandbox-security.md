# Sandbox Security

A sandbox is the execution boundary between an agent's tools and the host system. It controls what the agent can read, write, execute, and access on the network. Without a sandbox, a prompt injection or a buggy tool can wipe your disk, exfiltrate secrets, or run crypto miners on your machine.

## Why It Matters

Agents run untrusted code. They execute shell commands generated by an LLM, read files the model selects, and call APIs with model-chosen parameters. The model may be manipulated by adversarial inputs (prompt injection), hallucinate dangerous commands, or simply make mistakes. Sandboxing limits the blast radius: even if the agent does something wrong, the damage is contained.

## Trust Boundary Diagram

```
┌─────────────────────────────────────────────────┐
│                   HOST SYSTEM                    │
│                                                  │
│  ┌─────────────────────────────────────────────┐ │
│  │              SANDBOX BOUNDARY               │ │
│  │                                             │ │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  │ │
│  │  │  Agent   │  │  Tools   │  │  Files   │  │ │
│  │  │  Process │  │ Execution│  │ (scoped) │  │ │
│  │  └──────────┘  └──────────┘  └──────────┘  │ │
│  │                                             │ │
│  │  Allowed:                                   │ │
│  │  ✅ Read/write within /workspace            │ │
│  │  ✅ Run approved commands                   │ │
│  │  ✅ Network to API endpoints only           │ │
│  │                                             │ │
│  │  Blocked:                                   │ │
│  │  ❌ Access host filesystem outside /workspace│ │
│  │  ❌ Run arbitrary system commands            │ │
│  │  ❌ Access host network (127.0.0.1)         │ │
│  │  ❌ Install system packages                  │ │
│  └─────────────────────────────────────────────┘ │
│                                                  │
│  Host resources: filesystem, network, processes  │
└─────────────────────────────────────────────────┘
```

## Architecture 1: Docker Containers

The most common sandbox. Run each agent session in a container with limited resources.

### Dockerfile for Agent Sandbox

```dockerfile
# Dockerfile.agent-sandbox
FROM python:3.12-slim

# Create non-root user
RUN groupadd -r agent && useradd -r -g agent -m -d /home/agent agent

# Install common tools (no package manager in prod)
RUN apt-get update && apt-get install -y --no-install-recommends \
    git curl jq ripgrep \
    && rm -rf /var/lib/apt/lists/*

# Install Python dependencies
COPY requirements.txt /tmp/
RUN pip install --no-cache-dir -r /tmp/requirements.txt

# Security: remove package managers after setup
RUN apt-get purge -y --auto-remove && rm -rf /var/lib/apt/lists/*

# Set up workspace
WORKDIR /workspace
RUN chown agent:agent /workspace

# Drop to non-root
USER agent

# No CMD — orchestrator runs specific commands
```

### Launch Script

```bash
#!/bin/bash
# run-sandboxed-agent.sh

TASK="$1"
WORKSPACE="$(pwd)/workspace"

docker run --rm \
  --name "agent-$(date +%s)" \
  --user agent \
  --memory 2g \
  --cpus 1 \
  --pids-limit 100 \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=500m \
  --volume "$WORKSPACE:/workspace:rw" \
  --network none \
  --security-opt no-new-privileges \
  --security-opt seccomp=sandbox-profile.json \
  -e "OPENAI_API_KEY=$OPENAI_API_KEY" \
  agent-sandbox \
  python /workspace/agent.py --task "$TASK"
```

Key flags:
- `--memory 2g` — Prevent OOM from killing the host
- `--cpus 1` — Limit CPU usage
- `--pids-limit 100` — Prevent fork bombs
- `--read-only` — Root filesystem is immutable
- `--network none` — No network access (pass API key for model calls)
- `--no-new-privileges` — Can't escalate permissions

## Architecture 2: Firecracker microVMs

For stronger isolation than Docker. Firecracker VMs boot in ~125ms and provide hardware-level isolation.

```python
# Firecracker VM configuration (JSON API)
vm_config = {
    "boot-source": {
        "kernel_image_path": "/opt/kernels/vmlinux",
        "boot_args": "console=ttyS0 reboot=k panic=1 pci=off"
    },
    "drives": [{
        "drive_id": "rootfs",
        "path_on_host": "/opt/rootfs/agent-rootfs.ext4",
        "is_root_device": True,
        "is_read_only": True
    }],
    "machine-config": {
        "vcpu_count": 1,
        "mem_size_mib": 512
    },
    "network-interfaces": []  # No network
}

# Launch via Firecracker API
import requests
requests.put("http://localhost/vm", json=vm_config)
requests.put("http://localhost/actions", json={"action_type": "InstanceStart"})
```

**When to use Firecracker over Docker:**
- Multi-tenant SaaS where tenants don't trust each other
- Running user-provided code (not just model-generated)
- Compliance requirements demanding hardware isolation

## Architecture 3: WASM Sandboxes

WebAssembly sandboxes are the lightest option — microsecond startup, minimal overhead. Good for sandboxing individual tool executions.

```python
# Using wasmtime-py for WASM sandbox
from wasmtime import Engine, Store, Module, Instance, Linker

def run_in_wasm(wasm_path: str, function: str, args: list) -> str:
    """Execute a function in a WASM sandbox."""
    engine = Engine()
    store = Store(engine)

    # WASM modules have no filesystem/network access by default
    module = Module.from_file(engine, wasm_path)
    linker = Linker(engine)

    # Explicitly grant only what's needed
    # linker.define_wasi()  # Only if WASI access is required

    instance = linker.instantiate(store, module)
    result = instance.exports(store)[function](store, *args)
    return str(result)
```

## Permission Model

Define what each tool is allowed to do:

```python
from enum import Flag, auto
from pathlib import Path

class Permission(Flag):
    READ_FILES = auto()      # Read files in workspace
    WRITE_FILES = auto()     # Write files in workspace
    EXECUTE = auto()         # Run shell commands
    NETWORK = auto()         # Make HTTP requests
    SENSITIVE_FILES = auto() # Read .env, credentials
    SYSTEM_COMMANDS = auto() # apt-get, brew, sudo

# Tool permission registry
TOOL_PERMISSIONS = {
    "read_file":    Permission.READ_FILES,
    "write_file":   Permission.WRITE_FILES,
    "run_command":  Permission.EXECUTE,
    "web_search":   Permission.NETWORK,
    "web_fetch":    Permission.NETWORK | Permission.READ_FILES,
}

# Security levels
SECURITY_LEVELS = {
    "strict": Permission.READ_FILES,
    "standard": Permission.READ_FILES | Permission.WRITE_FILES | Permission.EXECUTE,
    "full": Permission.READ_FILES | Permission.WRITE_FILES | Permission.EXECUTE | Permission.NETWORK,
}

class Sandbox:
    def __init__(self, level: str = "standard", workspace: str = "/workspace"):
        self.allowed = SECURITY_LEVELS[level]
        self.workspace = Path(workspace).resolve()

    def check(self, tool_name: str, args: dict) -> bool:
        """Verify a tool call is within permissions."""
        required = TOOL_PERMISSIONS.get(tool_name, Permission(0))

        # Check permission flags
        if not (required & self.allowed) == required:
            raise PermissionError(
                f"Tool '{tool_name}' requires {required}, but level only allows {self.allowed}"
            )

        # Path containment check
        if "path" in args:
            target = Path(args["path"]).resolve()
            if not str(target).startswith(str(self.workspace)):
                raise PermissionError(
                    f"Path '{target}' is outside workspace '{self.workspace}'"
                )

        # Command blocklist
        if tool_name == "run_command":
            cmd = args.get("command", "")
            blocked = ["rm -rf /", "sudo", "curl | sh", "wget | bash", "chmod 777"]
            for pattern in blocked:
                if pattern in cmd:
                    raise PermissionError(f"Blocked command pattern: '{pattern}'")

        return True

# Usage in the tool loop
sandbox = Sandbox(level="standard", workspace="/workspace")

def safe_execute(tool_name: str, args: dict) -> str:
    try:
        sandbox.check(tool_name, args)
        return execute_tool(tool_name, args)
    except PermissionError as e:
        return f"🚫 Permission denied: {e}"
```

## Human-in-the-Loop for Destructive Actions

```python
DESTRUCTIVE_TOOLS = {"run_command", "write_file", "delete_file", "send_email"}

def execute_with_approval(tool_name: str, args: dict) -> str:
    """Require human approval for destructive actions."""
    if tool_name in DESTRUCTIVE_TOOLS:
        print(f"\n⚠️  Agent wants to: {tool_name}")
        print(f"   Args: {json.dumps(args, indent=2)}")
        approval = input("   Allow? [y/N]: ").strip().lower()
        if approval != "y":
            return "Action denied by user."

    return execute_tool(tool_name, args)
```

## Common Pitfalls

- **Docker ≠ security boundary** — Docker containers share the host kernel. A kernel exploit can escape. For true isolation against adversarial code, use Firecracker or gVisor.
- **Passing secrets as environment variables in Docker** — They're visible in `docker inspect`. Use Docker secrets or mount a read-only secrets file that's deleted after reading.
- **Forgetting network isolation** — An agent with network access can exfiltrate data, download malware, or call APIs you didn't intend. Default to `--network none` and whitelist specific endpoints.
